# Asset Metadata

The term "metadata" will refer to information that is extracted from the asset files themselves. In a future update, additional information will be generated by means of machine-learning, and that data will be categorized and stored separately.

## Overview

A similar application named PhotoPrism has a default gallery display mode in which assets (images and videos) are displayed in a grid. The asset thumbnails are fixed to 640 by 640 pixels and below the thumbnail there are several lines of metadata. The information for an image appears to be as follows:

- ML-assigned label / city / year
- user friendly date-time + timezone
- camera manufacturer and model
- lens manufacturer and model
- camera shutter speed, f-stop, ISO, focal length
- format (JPEG, MP4), dimensions, file size in KB or MB
- original file name
- city, region, country

Videos would include other fields, such as _duration_ and _codec type_.

Currently our application does not have all of that metadata. We will need a one-shot backfill to collect this information, and the import use case will need to collect this information going forward. The end goal is to expose this information via GraphQL and display the data in a new grid mode gallery display.

Note that the ML-assigned label is not available at this time, that will come later.

## Proposal

### GraphQL Fields

The `Asset` and `SearchResult` types in GraphQL will gain a nullable `metadata: AssetMetadata` field. `AssetMetadata` should be a typed object with optional fields rather than a generic list of name/value pairs — the field set is bounded and clients (the new gallery view) will want type-safety. Fields will include, all nullable:

- camera make, camera model
- lens make, lens model
- exposure time (as a string like `"1/250"`), f-stop, ISO, focal length in 35mm
- original date-time offset (e.g. `"+09:00"`)
- displayed width, displayed height (oriented dimensions; see Import section)
- GPS latitude, GPS longitude (raw decimal degrees — already extracted today and discarded after geocoding; cheap to retain)
- video-only: duration (seconds), frame rate (number), video codec name

Note the type is named `AssetMetadata`, not `Metadata`, to disambiguate from the future ML-derived metadata mentioned in the introduction.

The file format (JPEG, MP4, etc.) is derivable from `mediaType` and does not need to be stored.

### Database Changes

The default data store is CouchDB which stores documents in JSON format, allowing for additional fields. The alternate data store, PouchDB, is nearly identical to CouchDB. That leaves SQLite, which has an existing table named `assets`. Rather than expanding that table with numerous optional fields, we will create a new table named `metadata` with a foreign key to the asset record.

The `metadata` table (and the equivalent JSON sub-document for CouchDB/PouchDB) should also include a `raw` column/field that stores the full unparsed extractor output (EXIF tag map or `ffprobe` JSON). The cost is small (a few KB per asset) and it means future additions to `AssetMetadata` — say lens serial number — can be backfilled by a SQL/JSON update over already-fetched data rather than re-reading every blob.

Open question: the project does not appear to have an established SQLite migration framework. Adding the `metadata` table will require introducing one (or a one-off versioned bootstrap). Likewise, the CouchDB views in `couchdb-views.js` should be reviewed for any that would need to emit metadata fields, though most current views appear unaffected.

Response to open question: add a basic database migration framework as needed. The `couchdb-views.js` should not need to be changed, the search results are expected to remain unchanged.

### Import use case

The import use case will populate metadata at import time. We define an `AssetMetadata` entity that is a nullable property of `Asset`. Decision needed: does `putAsset` write both records (in a single SQLite transaction), or is there a separate `putMetadata` call that the use case orchestrates? Recommended: bundle in `putAsset` so callers cannot accidentally leave an asset without its metadata, and so SQLite can wrap both writes in one transaction.

For images, we will extract the following EXIF fields. Names below are the `ExifReader` tag names the codebase already uses — not the hex tag numbers, which the library does not expose directly:

- `OffsetTimeOriginal` — timezone offset for the original date-time
- `Make` — camera manufacturer
- `Model` — camera model
- `LensMake`
- `LensModel`
- `ExposureTime`
- `FNumber` — f-stop
- `ISOSpeedRatings`
- `FocalLengthIn35mmFilm` (preferred)
- `FocalLength` (only if `FocalLengthIn35mmFilm` is not available)
- `Orientation` — needed to compute displayed dimensions (see below)
- `PixelXDimension` / `PixelYDimension` (or fall back to the image's intrinsic dimensions)

Images and videos must record _width_ and _height_ in the **displayed** orientation so the column-based gallery layout receives correct aspect ratios. For images this means honoring the EXIF `Orientation` tag and swapping width/height for orientations 5, 6, 7, and 8. For videos, the `side_data_list` returned by `ffprobe` often contains a rotation matrix for portrait phone recordings; the stream's raw width/height must be swapped when rotation is ±90°.

For videos we will use `ffprobe` (see "Remote blob store" below for how this runs against Namazu). The fields of interest are:

- duration (`format.duration` or stream `duration`)
- frame rate (`r_frame_rate`, parsed from its rational form)
- video codec name (`codec_name` of the first video stream)
- displayed width / height (stream `width`/`height` adjusted for rotation)

Open decision: should we **drop `mp4box` in favor of `ffprobe` for `originalDate` extraction** during the import path? Today `helpers.ts:getCreationTime` uses `mp4box` because spawning a subprocess was undesirable. Once every import shells out to `ffprobe` anyway, that justification is gone, and unifying on one tool simplifies the code. Recommended: replace `mp4box` usage with `ffprobe` for new imports; leave the existing `getCreationTimeFromBlob` byte-range path alone since the backfill (per below) uses a different mechanism.

Latent bug worth fixing in the same pass: `helpers.ts:parseExifDate` constructs the original date with `Date.UTC(...)`, which treats local-clock EXIF fields as UTC. All existing `originalDate` values are off by the camera's timezone. Now that we capture `OffsetTimeOriginal`, the parse can honor it when available. We should either fix `originalDate` to use the offset, or explicitly leave it alone and document the discrepancy. Recommended: fix it, and re-run the existing `updateVideoDates` machinery to refresh stored dates.

To avoid three separate `ExifReader.load()` calls per image (one for the date, one for coordinates, one for the new metadata), the EXIF parsing in `helpers.ts` should be consolidated into a single pass that returns date, coordinates, and the metadata fields together.

### Query and resolver wiring

The storage of the search results should not change — embedding metadata in the search index would inflate both the on-disk size and the in-memory cache. Instead, the metadata will be fetched by the GraphQL resolvers and merged into the search results on the way to the client.

To avoid N+1 fetches, use a **request-scoped Apollo `DataLoader`** keyed by asset id rather than ad-hoc per-query batching. The loader resolves to `AssetMetadata` objects (or `null`), backed by a new `RecordRepository` method:

```typescript
fetchMetadata(assetIds: string[]): Promise<Map<string, AssetMetadata | null>>
```

The repository implementations:

- **CouchDB**: `db.fetch({ keys: [...] })` from the `nano` client.
- **PouchDB**: `db.allDocs({ keys: [...], include_docs: true })`. Note `allDocs` does not deduplicate the input array and includes deleted-stub entries; the implementation must filter those.
- **SQLite**: `SELECT ... FROM metadata WHERE asset_id IN (?, ?, ...)` (or `=ANY(...)` equivalents).

The client paginates search results (page size up to 72 in the web UI), so the loader fan-out is bounded.

### One-shot backfill

The existing database will be populated by a one-shot use case and GraphQL mutation, following the pattern of `update-video-dates.ts` / `updateVideoDates`. Open question: one mutation that handles images and videos together, or two (`backfillImageMetadata`, `backfillVideoMetadata`) so they can be retried independently? Recommended: two, since the video path is slower and more failure-prone (it has to talk to `ffprobe`/Namazu) and operators will want to re-run that half without re-touching images.

Backfill should store partial records: if some EXIF fields are present and others are missing, write what is available with nulls for the rest rather than skipping the asset entirely.

### Remote blob store (Namazu)

EXIF and `ffprobe` both need byte-level access to the file. For the local blob store this is trivial; for the Namazu remote blob store, we need a plan.

**Approach: extend Namazu with a metadata endpoint.** Add `GET /metadata/:blobId` to Namazu that returns the raw extractor output as JSON:

- For `video/*`: `ffprobe -v error -print_format json -show_format -show_streams <path>` and return stdout verbatim (parsed + re-serialized so extractor failures surface as 4xx). This is the canonical "dump useful info" invocation: `-show_format` gives container info (`format_name`, `duration`, `bit_rate`, format-level `tags` like `creation_time`), and `-show_streams` gives per-stream info (`codec_name`, `width`, `height`, `r_frame_rate`, `pix_fmt`, and a `side_data_list` that often carries the rotation matrix for portrait phone videos). Avoid `-show_packets`/`-show_frames` — they produce per-packet/per-frame dumps that are orders of magnitude larger.
- For `image/*`: read EXIF via whatever library Namazu uses (or `exiftool -json -n`) and return the full tag map. **Strip large embedded blobs** before returning (`Thumbnail`, `PreviewImage`, `MakerNote`) — these can be hundreds of KB of base64 and are useless to Tanuki.
- For other media types: return `204 No Content`.
- Cap response size (e.g. reject if >256KB after stripping) so a pathological file cannot return megabytes.

We return **raw extractor JSON, not a curated Tanuki-specific schema**, so that future additions to `AssetMetadata` do not require a Namazu redeploy. Tanuki picks the fields it wants from the raw response, and that same raw JSON is what gets stored in the `metadata.raw` column described above.

**Note:** the requested changes have been made to namazu.

The local blob store implementation of the metadata fetch can mirror this contract: produce the same JSON shape locally so the use-case-level code is identical regardless of backend.
